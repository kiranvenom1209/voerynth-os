Vœrynth OS – Developer Overview

This document explains how the Vœrynth OS app works and how it integrates with Home Assistant, for a developer who is new to the project.

1. Big picture
- Vœrynth OS is a React 18 + Vite single-page application.
- It is delivered as a web app, a Capacitor Android app, and an Electron desktop app.
- It runs on the local network and talks directly to a Home Assistant instance via WebSocket and REST.
- It is Layer 5 (“Neural Interface”) of a larger automation stack called Vœrynth Système.
- The UI is intentionally opinionated and minimal; most “intelligence” lives in lower layers (especially Vœrynth Core and Home Assistant).

2. System layering and role of Home Assistant
- The stack is organized into five layers:
  - Layer 1 – Hard Metal: physical connectivity (KNX, DALI, CAN/NMEA, I/O, gateways).
  - Layer 2 – Translation: protocols normalized into streams (e.g., MQTT/JSON, Signal K).
  - Layer 3 – State Engine: Home Assistant Core (integrations + state registry).
  - Layer 4 – Vœrynth Core: reasoning, policy, “stupidity filters”, auditability.
  - Layer 5 – Vœrynth OS: this UI, which presents state and accepts user intent.
- Home Assistant is treated as the integration and state layer, not the decision-maker.
- Vœrynth OS does not embed complex automation logic; it reflects state from HA and sends service calls back to HA.

3. How Vœrynth OS talks to Home Assistant
3.1 Low-level WebSocket client (HAConnection)
- File: src/services/HAConnection.js
- HAConnection is a small wrapper around Home Assistant’s WebSocket API at /api/websocket.
- It takes: url, token, onStateChange, onConnect, onDisconnect.
- It converts http/https URLs into ws/wss WebSocket URLs, then opens a socket to /api/websocket.
- On successful connection it authenticates with the long-lived token, fetches the full state, and subscribes to state_changed events.
- It maintains an in-memory map of entity states keyed by entity_id.
- Whenever HA sends new states or a state_changed event, HAConnection updates the map and calls onStateChange with the full state snapshot.
- It also tracks connection status and handles disconnects and reconnects.

3.2 HomeAssistantProvider context
- File: src/context/HomeAssistantContext.jsx
- HomeAssistantProvider is a React context provider that owns an HAConnection instance.
- It exposes, via useHomeAssistant():
  - connectionStatus (connecting / connected / disconnected),
  - hassStates (entity_id → state objects),
  - actions: connect(url, token, callbacks?), disconnect(), callService(domain, service, data),
  - helpers: getSavedCredentials(), shouldAutoConnect(), etc.
- On connect(), it:
  - creates HAConnection with the current URL and token,
  - wires onStateChange to set React state hassStates,
  - updates connectionStatus when connected or disconnected,
  - persists the URL and token in local storage or Capacitor storage,
  - exposes the HAConnection instance on window.__haConnection for advanced usage.
- Components can subscribe to HomeAssistantContext to get live HA state and to call HA services.

3.3 High-level HA client and global HA store
- File: src/services/haClient.js
- haClient uses the home-assistant-js-websocket library.
- It abstracts:
  - connect(url, token) – sets up a WebSocket connection with proper auth,
  - callWS(type, params) – send arbitrary WebSocket commands,
  - callREST(path, options) – call /api/* REST endpoints with the bearer token,
  - error normalization so UI code can handle common HA failure modes.
- File: src/stores/haStore.js
- useHAStore is a Zustand store that mirrors HA’s own frontend data model.
- It keeps:
  - statesByEntityId – entity state map,
  - areas, devices, entityRegistry, configEntries – fetched via haClient.callWS("config/..."),
  - derived maps areasById, devicesById, entityRegByEntityId for fast lookups.
- The initialize(hassStates) method:
  - takes the current live hassStates from HomeAssistantContext,
  - uses them as the initial entity state snapshot,
  - fetches areas, devices, entity registry, and config entries in parallel via HA WebSocket commands,
  - builds HA-style join data so the UI can know which entities belong to which devices and areas.
- The updateStates(hassStates) method is called when HomeAssistantContext updates, keeping the store in sync with live HA state.

4. App structure and startup flow
- File: src/App.jsx
- App.jsx wraps the main content in two providers:
  - AccentColorProvider – manages accent color choices for Tailwind-based styling.
  - HomeAssistantProvider – provides HA connection and entity state.
- AppContent handles:
  - splash screen / branding sequence,
  - loading and applying saved HA credentials,
  - opening the configuration modal when no credentials are present or connection fails,
  - rendering the main dashboard, Devices & Entities view, Settings, etc.
- On mount, AppContent:
  - calls getSavedCredentials() from HomeAssistantContext,
  - if there is a saved URL and token and the user did not manually disconnect, it auto-connects,
  - otherwise, it shows the configuration view so the user can enter HA URL and token.

5. Example data flow: toggling a light
- User taps a light toggle in the UI.
- The React component calls callService("light", "turn_on", { entity_id: "light.bedroom" }) via HomeAssistantContext or haClient.
- Home Assistant receives the call_service request, executes it, and updates the entity state internally.
- HA emits a state_changed event over the WebSocket connection for that entity.
- HAConnection receives the event, updates its internal states map, and calls onStateChange with the updated states.
- HomeAssistantProvider updates its hassStates React state and notifies any subscribers.
- useHAStore.updateStates(hassStates) keeps the global HA store in sync.
- Any component that is reading the light’s state via hassStates or useHAStore re-renders, showing the new state (on/off).

6. Main UI surfaces built on top of HA
- Dashboard:
  - Landing view that shows a header, quick modes, and a set of summary cards (neural interface status, weather, transit, energy, residents, ecosystem health).
  - Reads HA state and possibly derived data from useHAStore.
  - Quick modes are global modes (e.g., “At Sea”, “Guests Aboard”) that drive coordinated changes via HA services.
- Devices & Entities view:
  - Backed by useHAStore and its HA-style joins.
  - Presents Devices, Entities, and Areas in a single combined screen with tabs and search.
  - Provides inline edit dialogs for entities, devices, and areas; these write back to HA via WebSocket commands.
  - Includes a debug panel with subscription state, counts, and loading/error information for operators.
- Settings module:
  - Implements views for areas, devices/services, entities, automations, integrations, people, system, and add-ons.
  - Allows most day-to-day HA configuration to be done directly from Vœrynth OS instead of the HA web UI.
- Voice and wake-word:
  - On supported platforms, a WakeWordService listens for a wake phrase.
  - When the phrase is detected, the app calls an Assist / voice conversation service in HA (assist_satellite.start_conversation).
  - From HA’s perspective this is just another service call; HA handles the voice pipeline.

7. What you need to run the app
- Infrastructure:
  - A Home Assistant instance reachable on the same LAN as the device running Vœrynth OS.
  - A long-lived access token created from the HA user profile.
  - Optional: configured Assist / voice satellite if you want wake-word flows.
- App:
  - Node.js 18+ and npm installed.
  - From the repo root:
    - npm install
    - npm run dev
  - Open http://localhost:5173 in a browser.
  - On first run the app will prompt for HA URL and token and then attempt to connect.

8. Summary
- Vœrynth OS is a Layer 5 “command deck” UI that sits on top of Home Assistant.
- It uses:
  - HAConnection for low-level WebSocket state and service calls,
  - HomeAssistantProvider for React context and connection lifecycle,
  - haClient plus useHAStore for registries and HA-style join logic,
  - React + Vite + Tailwind + Zustand for UI and state management.
- It does not replace Home Assistant’s automation engine; instead, it presents a curated, role-focused interface that reads HA state and sends HA service calls on behalf of users.
